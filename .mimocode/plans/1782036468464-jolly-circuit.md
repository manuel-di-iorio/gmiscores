| Layer | Tecnologia |
|-------|-----------|
| Framework | Next.js 14+ (App Router) |
| Lingua | TypeScript |
| ORM | Prisma (MySQL) |

## Fase 2: Autenticazione (giorno 4-6)

### NextAuth.js + Discord
- Configurare provider Discord con le stesse env vars (`DISCORD_CLIENT_ID`, `DISCORD_CLIENT_SECRET`)
- Callback URL: `/api/auth/callback/discord`
- Callback custom per approval check:
  - Dopo login, verificare `user.approved === true`
  - Se non approvato, reindirizzare a `/approval`
  - Se approvato, creare sessione

### Cookie cifrati
- `iron-session` con le stesse chiavi di `lib/crypt.php`
- Cookie `user_id` cifrato con AES + HMAC
- Scadenza: 1 anno (come il PHP)

### Middleware
```typescript
// middleware.ts
// 1. Route protette: redirect a /login se non autenticato
// 2. Maintenance check: blocca scritture
// 3. Theme: legge cookie, applica classe .dark
// 4. Locale: legge cookie lang, imposta header Accept-Language
```

### Sessione team
- Cookie `selected_team_id` gestito dal middleware
- Endpoint per cambiare team: `/api/switch-team`

### File da portare
- `lib/crypt.php` → `lib/crypto.ts` (Node.js `crypto` module)
- `lib/checkSession.php` → middleware
- `lib/csrf.php` → `lib/csrf.ts`
- `api/v1/gmi-login.php` → NextAuth Discord provider

**Verifica:** Login Discord funziona, cookie cifrato viene imposto, route protette reindirizzano, theme persiste.

## Fase 3: Database e Modelli (giorno 7-10)

### Prisma Client
- Generare client: `npx prisma generate`
- Singleton pattern in `lib/db.ts`

### Modelli — portare 7 file PHP → TypeScript

**Per ogni modello:**
1. Copiare la struttura del modello PHP
2. Convertire le query MySQL in Prisma API (o raw queries per query complesse)
3. Aggiungere type safety con TypeScript

**Priorità:**
1. `User` — semplice, serve subito per auth
2. `Game` — serve per la maggior parte delle pagine
3. `Team` + `TeamMember` — serve per il team context
4. `Leaderboard` — serve per la gestione game
5. `Player` — serve per scores
6. `Ban` — serve per score management
7. `Score` — il più complesso, portarlo per ultimo

**Score.php (674 righe):**
- ~40 metodi statici da convertire
- Query complesse: ranking, statistiche, grafici, filtro, export
- Usare Prisma `findMany` con `include` e `where` dinamico
- Per query molto complesse: `prisma.$queryRaw`

**insertScore.php:**
- Logica insert-mode (higher/lower/all)
- Calcolo rank post-insert
- Verifica ban prima dell'insert
- Hash verification
- Portare come Server Action

### Seed database
- Script per popolare DB di test con dati realistici

**Verifica:** Tutti i modelli funzionano con Prisma, query di lettura/scrittura testate con dati seed.

## Fase 10: API Pubbliche e Rate Limiting (giorno 31-32)

### API v1 (per GameMaker SDK)
```typescript
// /api/v1/add/route.ts
// POST: score submission
// Headers: Access-Control-Allow-Origin: *
// Rate limit: 10 req/60s per IP
// Hash verification: sha1(game_data + client_secret)
// Cloudflare IP detection

// /api/v1/list/route.ts
// GET: score listing
// Rate limit: 60 req/60s per IP
// Private leaderboard hash auth
```

### Rate Limiting
- `@upstash/ratelimit` con Redis, oppure mantenere MySQL sliding window
- Headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `Retry-After`

### CORS
- `Access-Control-Allow-Origin: *` per le API pubbliche
- Configurare in Next.js middleware o route handlers

**Verifica:** SDK GameMaker funziona con le nuove API, rate limiting attivo, CORS headers presenti.
